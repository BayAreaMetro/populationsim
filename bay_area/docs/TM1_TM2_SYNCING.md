# TM1 + TM2 Syncing

This document tracks the two-phase process for syncing TM1 and TM2 population synthesis onto a shared codebase and eventually into a single modernized pipeline.

**What was done:** The `tm2` branch now contains both the TM1 BAT-based workflow and the TM2 Python pipeline in one place. Merging `master` into `tm2` is complete. Both workflows are code-complete and ready to test.

**Phase 1 (current):** Run and validate both TM1 and TM2 on the `tm2` branch. Confirm each produces results matching the respective prior reference runs. Once both pass, merge `tm2` into `master` — making `master` the single production branch for both model types.

**Phase 2 (after merge to master):** Modernize TM1 — replace the R script dependency and manual file copies with a Python pipeline, so TM1 runs end-to-end from a single command like TM2 already does.

---

## 1. How to Run TM1

TM1 runs exactly as it always did: via `run_populationsim.bat`. No Python pipeline involved.

### Prerequisites

1. **PUMS data** — 2017–2021 5-year ACS PUMS pre-downloaded to:
   ```
   M:\Data\Census\PUMS_2021_5Year_Crosswalked\
     bay_area_households_2017_2021_crosswalked.csv
     bay_area_persons_2017_2021_crosswalked.csv
   ```

2. **TAZ control files** — generated by R scripts in `travel-model-one/utilities/taz-data-baseyears/`:
   - Copy `TAZ1454 YYYY Popsim Vars.csv` to `hh_gq/data/taz_summaries.csv`
   - Copy `TAZ1454 YYYY Popsim Vars County.csv` to `hh_gq/data/county_marginals.csv`
   - *(The BAT does this copy automatically for base years 2015, 2020, 2023 if `TMPATH` is set correctly)*

3. **Crosswalk and configs** — already committed to this repo:
   - `hh_gq/data/geo_cross_walk_tm1.csv`
   - `hh_gq/configs_TM1/` (settings.yaml, controls.csv, logging.yaml)

### Run

```bat
:: Edit run_populationsim.bat -- top lines:
set MODELTYPE=TM1
set TMPATH=X:\travel-model-one-master\utilities\taz-data-baseyears
set BAUS_RUNNUM=run182
set OUTPUT_SUFFIX=STIP2026_RTP2021FBP_20251113

:: Then run:
run_populationsim.bat 2023
```

### What the BAT does (TM1 mode)

| Step | Script | What it does |
|------|--------|-------------|
| 1 | `create_seed_population.py --model_type TM1` | 2017-2021 PUMS -> `seed_households.csv`, `seed_persons.csv`; income in `hh_income_2000` |
| 2 | Copy TAZ controls | `TAZ1454 YYYY Popsim Vars.csv` -> `hh_gq/data/taz_summaries.csv` |
| 3 | `add_hhgq_combined_controls.py --model_type TM1` | Derives `numhh_gq`, `hh_size_1_gq` -> `taz_summaries_hhgq.csv` |
| 4 | `run_populationsim.py --config hh_gq/configs_TM1` | PopulationSim IPF (TAZ geography) |
| 5 | `postprocess_recode.py --model_type TM1` | Renames columns to TM1 format: `HINC`, `UNITTYPE`, `VEHICL`, `PERSONS`, `hinccat1` |

### TM1 Outputs

```
<OUTPUT_DIR>/
  synthetic_households.csv        (raw PopulationSim output)
  synthetic_persons.csv
  synthetic_households_recode.csv (HHID, TAZ, HINC, hworkers, VEHICL, BLD, TEN, PERSONS, HHT, UNITTYPE, hinccat1, pct_of_poverty, ...)
  synthetic_persons_recode.csv    (HHID, PERID, AGE, SEX, pemploy, pstudent, ptype)
  summary_melt.csv
  final_summary_TAZ.csv
  final_summary_COUNTY_[1-9].csv
  taz_summaries.csv               (saved input)
  taz_summaries_hhgq.csv          (saved input)
```

## 2. How to Run TM2

TM2 uses the Python pipeline.

### Prerequisites

1. **PUMS data** — 2019-2023 5-year ACS PUMS at:
   ```
   M:\Data\Census\PUMS_2023_5Year_Crosswalked\
     bay_area_households_2019_2023_crosswalked.csv
     bay_area_persons_2019_2023_crosswalked.csv
   ```
   *(or run `python tm2_pipeline.py pums` to download)*

2. **Crosswalk and configs** — already committed:
   - `hh_gq/data/geo_cross_walk_tm2_maz.csv`
   - `hh_gq/configs_TM2/`

3. **Census API key** at `M:\Data\Census\API\new_key\`

### Run

```bash
cd bay_area
conda activate popsim

# Full pipeline (~2-3 hours):
python tm2_pipeline.py full --force

# Or step by step:
python tm2_pipeline.py seed --force           # create_seed_population.py (TM2 mode)
python tm2_pipeline.py controls --force       # create_baseyear_controls.py
python tm2_pipeline.py populationsim --force  # run_populationsim.py (configs_TM2)
python tm2_pipeline.py postprocess --force    # postprocess_recode.py --model_type TM2
python tm2_pipeline.py analysis --force       # run_all_summaries.py
```

### TM2 Outputs

```
output_2023/populationsim_working_dir/output/
  synthetic_households.csv
  synthetic_persons.csv
  households_2023_tm2.csv         (final, TM2 format)
  persons_2023_tm2.csv
output_2023/charts/               (analysis charts and dashboards)
```

## 3. Key Differences Between TM1 and TM2

| | TM1 | TM2 |
|---|---|---|
| **Entry point** | `run_populationsim.bat MODELTYPE=TM1` | `python tm2_pipeline.py full` |
| **Finest geography** | TAZ (1,454 zones) | MAZ (~27,500 zones) |
| **Controls source** | R scripts in `travel-model-one` repo, copied in manually | `create_baseyear_controls.py` (Census API, fully automated) |
| **PUMS vintage** | 2017-2021 (2010-vintage PUMA codes) | 2019-2023 (2020-vintage PUMA codes) |
| **Income field** | `hh_income_2000` -> output `HINC` (2000 dollars) | `hh_income_2010` -> output `HHINCADJ` (2010 dollars) |
| **Income bins** | 4 bins: HHINCQ1-4 ($30k/$60k/$100k in 2000$) | Continuous (8-bin controls internally) |
| **Age bins** | 5 bins (0-4, 5-19, 20-44, 45-64, 65+) | 4 bins |
| **Unit type field** | `UNITTYPE` | `TYPE` |
| **Workers field** | `hworkers` | `NWRKRS_ESR` |
| **PopulationSim config** | `hh_gq/configs_TM1/` | `hh_gq/configs_TM2/` |
| **Geo crosswalk** | `geo_cross_walk_tm1.csv` (PUMA->TAZ->COUNTY) | `geo_cross_walk_tm2_maz.csv` (PUMA->TAZ->MAZ->COUNTY) |
| **TM1-only outputs** | `hinccat1`, `pct_of_poverty`, `poverty_income_*` | Not produced |

## Phase 1: Test on tm2 Branch, then Merge to Master

Both workflows are code-complete on the `tm2` branch. The gate to merging into `master` is a passing test run for each.

### Step 1 — Test TM1 on the tm2 branch

Reference: the most recent known-good TM1 run on `master` (e.g. `STIP2026_RTP2021FBP_20251113`).

```bat
:: Force a clean rebuild of the seed population
del hh_gq\data\seed_households.csv
del hh_gq\data\seed_persons.csv

:: Set MODELTYPE=TM1 in run_populationsim.bat, then:
run_populationsim.bat 2023
```

**Validation checks:**

| Check | File / column | Expected |
|-------|--------------|----------|
| Seed PUMA codes | `seed_households.csv`, `PUMA` column | 2010-vintage codes (5-digit, e.g. `01101`) |
| Seed income | `seed_households.csv`, `hh_income_2000` | Non-zero; Bay Area median ~$65k-$80k |
| GQ combined controls | `taz_summaries_hhgq.csv`, `numhh_gq` | `TOTHH + gq_tot_pop` for every row |
| Synthesis total HH | `synthetic_households_recode.csv`, row count | Matches sum of `TOTHH` in `taz_summaries.csv` |
| `HINC` range | `synthetic_households_recode.csv` | No zeros for regular HH (`UNITTYPE==1`) |
| `hinccat1` | `synthetic_households_recode.csv` | 1-4 only |
| `UNITTYPE` | `synthetic_households_recode.csv` | 1 (HH) or 3 (GQ) only |
| Output vs reference | diff against master run | Row counts and distribution statistics match |

### Step 2 — Test TM2 on the tm2 branch

Reference: last successful `output_2023` run before TM1+TM2 syncing work began.

```bash
python tm2_pipeline.py full --force
```

**Validation checks:**

| Check | Expected |
|-------|----------|
| `households_2023_tm2.csv` row count | ~2,958,470 |
| `persons_2023_tm2.csv` row count | ~7,563,557 |
| All controls within tolerance | `python run_all_summaries.py` produces no red flags |
| `HHINCADJ` populated | No zeros for regular HH |
| Income distribution | Median ~$95k-$120k (2010 dollars) |

### Step 3 — Merge tm2 into master

Once both tests pass:

```bash
git checkout master
git merge tm2 --no-ff -m "Sync TM1 + TM2: add TM2 Python pipeline and dual-mode scripts to master"
git push origin master
```

From this point, `master` is the single production branch for both TM1 and TM2 runs.

## Phase 2: Modernize TM1

After the merge to master, the next goal is to make TM1 fully self-contained in Python — one command, no R scripts, no manual file copies — matching what TM2 already does.

### What needs to be replaced

| Current requirement | Target |
|---------------------|--------|
| Run R scripts in `travel-model-one` repo | `create_baseyear_controls.py` extended to produce TAZ-level controls natively |
| Manual copy of `TAZ1454 YYYY Popsim Vars.csv` | Pipeline fetches from Census API internally |
| `add_hhgq_combined_controls.py` called by BAT | Integrated into pipeline as a numbered step |
| `run_populationsim.bat` as entry point | `python tm2_pipeline.py full --model_type TM1 --year YYYY` |

### Task list

| # | Task | Effort |
|---|------|--------|
| 2.1 | Add `--model_type TM1` to `tm2_pipeline.py` -- route to `configs_TM1`, skip MAZ controls, call `add_hhgq_combined_controls.py --model_type TM1` | Medium |
| 2.2 | Validate `taz_summaries.csv` exists or error with helpful message pointing to `travel-model-one` | Small |
| 2.3 | Port R ACS pulls to Python: B19001 (income), B25009 (HH size), B08202 (workers), B01001 (age) | Medium |
| 2.4 | Port CPI deflation: convert ACS nominal income -> 2000 dollars using BLS CPI | Small |
| 2.5 | Port income rebinning: ACS categories -> HHINCQ1-4 ($30k/$60k/$100k thresholds in 2000$) | Small |
| 2.6 | Port GQ pull: Decennial PCT19 -> `gq_type_univ`, `gq_type_mil`, `gq_type_othnon` by TAZ | Medium |
| 2.7 | Aggregate block-group / tract data to TAZ using TM1 TAZ-block crosswalk | Medium |
| 2.8 | Write `hh_gq/data/taz_summaries.csv` in same format `add_hhgq_combined_controls.py` expects | Small |
| 2.9 | Update `run_all_summaries.py` to skip MAZ-only scripts when `--model_type TM1` | Small |
| 2.10 | End-to-end test: compare Python-produced TM1 controls to R-produced controls for 2023 | Validation |

**End state:** `python tm2_pipeline.py full --model_type TM1 --year 2023` -- single command, no R dependency, no manual file copy, same outputs as current BAT workflow.
